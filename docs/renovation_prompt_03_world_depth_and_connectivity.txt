# Premium Game World Depth, Connectivity, Terrain, and Building Renovation Prompt

================================================================================
PURPOSE OF THIS PROMPT
================================================================================

This is a reusable renovation prompt for transforming an existing HTML /
JavaScript / CSS game into a richer, more connected, more explorable, more
interactive world.

This prompt is especially intended for games that already have a playable
standalone mode but still feel like a sequence of disconnected rectangular areas,
flat backgrounds, or empty arenas.

The goal is to improve the game world itself:

- connected world layout
- smooth transitions between areas
- preview/buffer zones at area edges
- terrain features
- collision-safe map design
- trees, grass, rocks, cliffs, rivers, ponds, hills, roads, bridges, houses,
  castles, ruins, towns, caves, and interiors
- enterable buildings
- destroyable/gatherable objects
- mining and resource collection
- crafting/building systems
- better area connectivity
- spawn safety
- pathing safety
- map readability
- player movement through believable spaces
- better use of light and darkness
- better world assets and object variety
- bug prevention for stuck states at entrances, borders, doors, caves, and
  transitions

This prompt is not primarily about GUI polish. It is about making the world feel
real, connected, explorable, interactive, and mechanically useful.

================================================================================

## Mission

You are an expert game systems designer, world designer, level designer,
JavaScript engineer, creative technologist, and gameplay programmer.

Your mission is to renovate the world and interaction systems of an existing
HTML / JavaScript / CSS game.

The renovated game should feel less like a set of isolated screens and more like
a continuous world made of connected places.

The player should be able to:

- travel smoothly between areas
- see hints/previews of neighboring areas before entering them
- avoid getting stuck at borders or entrances
- move around real terrain features
- enter buildings, caves, fortresses, houses, shops, ruins, and dungeons
- interact with trees, rocks, grass, ores, crates, barrels, bridges, doors,
  gates, signs, and other world objects
- destroy or mine certain objects
- collect resources
- use resources for crafting or building
- understand where they are through an in-game map
- experience areas with distinct visual identity, lighting, terrain, and
  gameplay function

The goal is not to replace the original game. The goal is to evolve it into a
deeper and more connected version of itself.

================================================================================

## Non-Negotiable Philosophy

1. The world must be connected.
   Do not treat each screen as an unrelated rectangle. Areas should connect
   spatially and logically.

2. Transitions must be smooth and safe.
   The player should never wait several seconds at an edge, hard cut into an
   unrelated space, or spawn inside a wall.

3. Preview zones must match destination collision.
   If the next area has a wall, cliff, river, gate, building, or blocked path at
   the transition point, the preview/buffer area must show that obstruction too.
   Never show an open path that leads to a blocked spawn.

4. The player must not get stuck.
   Every entrance, exit, door, cave mouth, bridge, fortress gate, interior
   portal, and border transition must be spawn-safe and escape-safe.

5. Terrain must matter.
   Trees, rocks, rivers, walls, cliffs, ponds, buildings, and ruins should affect
   navigation, combat, exploration, resources, or progression.

6. Assets should be functional when possible.
   A rock can be decoration, but some rocks should be mineable. A tree can be
   scenery, but some trees should provide wood. A house can be background, but
   some buildings should be enterable.

7. Systems should connect.
   Mining gives stone or ore. Trees give wood. Resources feed crafting. Crafting
   creates tools, structures, repairs, upgrades, or unlocks.

8. Darkness is a world property, not a browser theme.
   Outdoor meadows, towns, roads, and forests can be bright and sunny. Caves,
   evil fortresses, haunted zones, dungeons, and night areas can be dark.

9. Prefer data-driven world content.
   Regions, transitions, biomes, terrain, props, resources, interiors, and
   collision should be defined in reusable structures where practical.

10. Avoid decorative-only fake systems.
    Do not add dead buttons, fake building menus, non-working mines, non-enterable
    doors labeled as enterable, or placeholder resources with no use.

## Step 1 — Understand the Current World

Before changing code, produce a written assessment of the current game world.

--------------------------------------------------------------------------------
1.1 Core World Identity
--------------------------------------------------------------------------------

Answer:

- What kind of game world is this?
- Is it an overworld, dungeon crawler, RPG map, arena sequence, survival world,
  sandbox, town-and-dungeon structure, or something else?
- How does the player currently move between areas?
- Are areas spatially connected or just swapped?
- Is there a world map or region graph?
- Are there meaningful landmarks?
- Are there enterable locations?
- Are there obstacles?
- Are there resources?
- Are there buildable/craftable systems?
- What is missing from the sense of place?

--------------------------------------------------------------------------------
1.2 Current Area Transition Model
--------------------------------------------------------------------------------

Document:

- How the current area is represented.
- How area IDs/region IDs are stored.
- How edge transitions are triggered.
- Whether transition is based on absolute screen edge, margin, portal, door, or
  coordinate.
- Whether transitions are instant, delayed, animated, or asynchronous.
- Whether transitions can overlap.
- Whether the player can be placed into invalid positions.
- Whether the next area is previewed before entry.
- Whether transition collision data is shared between adjacent regions.
- Known bugs, including stuck states at entrances or borders.

Specifically check for problems like:

- waiting too long at a border
- transition requiring exact edge contact
- hard cuts to unrelated spaces
- player spawning inside walls
- walls too close to entrance spawn points
- missing escape routes
- one-way transitions that trap the player
- preview area not matching the next area's real collision
- region data that visually suggests a path but blocks the player after loading

--------------------------------------------------------------------------------
1.3 Current Map Connectivity
--------------------------------------------------------------------------------

Document:

- Whether the map is a grid, graph, list, tiled map, random generation, or
  hardcoded screens.
- Whether each area has neighbors.
- Whether coordinates are world-space or screen-space.
- Whether logical regions span multiple visible blocks.
- Whether entrances/exits have explicit spawn points.
- Whether interiors are separate maps or part of the same world.
- Whether there is a concept of discovered regions.
- Whether the player can backtrack safely.

--------------------------------------------------------------------------------
1.4 Current Terrain and Collision
--------------------------------------------------------------------------------

Document:

- What terrain exists now.
- What objects exist now.
- Whether objects have collision.
- Whether collision is tile-based, rectangle-based, pixel-based, entity-based,
  or absent.
- Whether enemies respect collision.
- Whether projectiles/sword swings respect terrain.
- Whether terrain creates meaningful routes.
- Whether terrain blocks transitions accidentally.
- Whether collision can be inspected/debugged.

Look for:

- trees
- rocks
- cliffs
- rivers
- ponds
- hills
- walls
- fences
- houses
- castles
- gates
- bridges
- caves
- ruins
- doors
- signs
- mineable nodes
- destructible props
- resource objects
- buildable structures

--------------------------------------------------------------------------------
1.5 Current Resource and Building Systems
--------------------------------------------------------------------------------

Document:

- Are there resources such as wood, stone, ore, gems, herbs, food, fiber, gold,
  crystals, or relics?
- Can resources be gathered?
- Are any world objects destroyable?
- Are tools required?
- Is there mining, chopping, harvesting, digging, or salvaging?
- Are resources used for crafting?
- Can the player build anything?
- Are built objects persisted?
- Are there placement rules?
- Can building create blocked paths or stuck states?
- Can the player remove or move built objects?

--------------------------------------------------------------------------------
1.6 Current Lighting and Biome Identity
--------------------------------------------------------------------------------

Document:

- Which areas are bright?
- Which areas are dark?
- Is darkness applied globally?
- Are outdoor areas incorrectly dark?
- Are caves/fortresses/dungeons appropriately dark?
- Are biomes visually distinct?
- Does lighting affect gameplay or only appearance?
- Are UI colors incorrectly influencing world colors?

## Step 2 — Identify World-Depth Leaps

Critically identify where the game world falls short. Group findings into the
following categories.

================================================================================
2.1 Connectivity Leap
================================================================================

Ask:

- Does the world feel continuous?
- Can the player understand how one place relates to another?
- Are neighboring areas spatially consistent?
- Are transitions fast, readable, and safe?
- Are entrances/exits designed, or are they just screen edges?
- Can the player get stuck because the source and destination do not agree?

Improve by adding:

- explicit region graph
- world-space coordinates
- region coordinates
- neighbor links
- transition zones
- entrance markers
- safe spawn points
- collision-aware transition validation
- preview/buffer strips that use destination terrain
- smooth camera panning or view interpolation
- multi-block regions instead of single-screen rectangles
- connected roads, rivers, walls, bridges, and gates

================================================================================
2.2 Transition Safety Leap
================================================================================

Ask:

- Can the player transition into a wall?
- Is the destination spawn point guaranteed walkable?
- Does the source preview show the destination boundary truthfully?
- Are there escape paths after entering?
- Are doorways and cave mouths wide enough?
- Are fortress gates offset from walls?
- Are walls too close to transition spawn points?

Implement:

- `validateTransition(source, target, direction)`
- `findSafeSpawn(targetRegion, entranceId, fallbackPosition)`
- collision-aware spawn adjustment
- minimum spawn clearance radius
- blocked entrance detection
- transition debug overlay if useful
- mirrored source/destination edge collision
- preview collision consistency checks
- guard rails against impossible transitions

Special requirement:

If a destination entrance is blocked by a wall, cliff, river, building, or other
solid object, the source-side preview/buffer zone must also show that obstruction
and prevent transition through that blocked segment.

## Step 5 — Execute the Renovation

Implement working systems. Do not stop at decorative additions.

================================================================================
5.1 World Model
================================================================================

Create or improve a world model where regions are connected by real metadata.

Each region should support fields such as:

- id
- name
- coordinates
- size
- biome
- lighting
- neighbors
- transition zones
- terrain
- props
- resource nodes
- enemy spawns
- buildings
- interiors
- points of interest
- discovered state
- safe spawn points

Example structure:

```js
{
  id: "sunny-meadow",
  name: "Sunny Meadow",
  grid: { x: 0, y: 0 },
  size: { columns: 2, rows: 2 },
  biome: "meadow",
  lighting: "bright",
  neighbors: {
    north: "forest-road",
    east: "river-crossing",
    south: "village-green"
  },
  transitions: [
    {
      id: "east-road",
      direction: "east",
      targetRegion: "river-crossing",
      sourceZone: { x: 920, y: 220, w: 80, h: 160 },
      targetSpawn: { x: 96, y: 300 },
      requiredClearance: 32
    }
  ],
  terrain: [],
  props: [],
  resources: [],
  buildings: [],
  enemySpawns: []
}
