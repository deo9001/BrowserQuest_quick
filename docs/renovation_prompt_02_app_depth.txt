# Premium App/Game Systems Renovation Prompt

================================================================================
PURPOSE OF THIS PROMPT
================================================================================

This is a reusable, project-agnostic renovation prompt for transforming any
existing HTML / JavaScript / CSS application into a deeper, richer, more complete,
more replayable, more satisfying, and more substantial application.

Unlike a UI-first renovation prompt, this version focuses primarily on the app
itself: the actual game mechanics, creative systems, simulation depth, content,
rules, progression, data model, player goals, runtime behavior, and long-term
engagement.

If the target app is a game, improve the game itself:
- graphics and visual effects
- world design
- enemies
- enemy AI
- boss fights
- quests
- items
- weapons
- armor
- tools
- mounts
- leveling systems
- crafting
- resources
- achievements
- secrets
- locations
- procedural content
- replayability
- save/load
- difficulty curves
- moment-to-moment fun

If the target app is not a game, apply the same philosophy to its actual domain:
make the core product more capable, more intelligent, more useful, more
expressive, more complete, and more durable — not merely prettier.

================================================================================

## Mission

You are an expert software architect, game designer, systems designer,
front-end engineer, creative technologist, and product builder.

Your mission is to take an existing HTML / JavaScript / CSS app and renovate the
substance of the application. Go far beyond cosmetic improvements. Identify what
the app is truly trying to be, then make it dramatically more complete.

The renovated app should feel like a fuller version of itself:
- If it is a game, make it more fun, deeper, more replayable, and more alive.
- If it is a tool, make it more powerful, capable, automated, and useful.
- If it is a creative app, make it more expressive, flexible, and exportable.
- If it is a simulator, make the simulation richer and more meaningful.
- If it is a dashboard, make the data model, insights, and workflows stronger.
- If it is an educational app, make the learning progression deeper and clearer.

The goal is not merely to improve the interface. The goal is to improve what the
app can actually do.

================================================================================

## Non-Negotiable Philosophy

1. Improve the app itself, not only its presentation.
   A prettier app with the same shallow behavior is not enough. Add meaningful
   mechanics, content, workflows, state, intelligence, and replay value.

2. Go above and beyond.
   Do not implement the minimum obvious change. Anticipate what would make the
   app feel like a more complete product and build it.

3. Bigger and richer is welcome.
   Do not equate quality with minimalism. More systems, more content, more
   mechanics, more data, more modes, more levels, more entities, and more
   progression are valuable when implemented coherently.

4. Preserve and extend the original identity.
   Do not replace the app with an unrelated app. Understand what it already is,
   then evolve it into a much more complete version of itself.

5. Prefer playable, usable systems over decorative mockups.
   If adding a feature, make it actually work. Do not add fake buttons,
   placeholder menus, dead content, or non-functional claims.

6. Design for repeated use.
   The app should remain interesting and stable after multiple runs, multiple
   sessions, repeated resets, and edge-case interactions.

7. Content matters.
   If the app is a game, add real content: enemies, items, areas, powers, quests,
   rewards, hazards, bosses, upgrades, secrets, and progression. Do not stop at
   UI framing.

8. Systems should connect.
   Items should affect combat. Levels should unlock abilities. Quests should
   reward useful things. Enemies should interact with the world. Tools should
   change what the player can do. Avoid isolated features that do not matter.

## Step 1 — Understand the Current App

Before changing code, produce a written assessment of the current app.

--------------------------------------------------------------------------------
1.1 Core Identity
--------------------------------------------------------------------------------

- What is this app fundamentally?
- Is it a game, tool, editor, simulator, toy, dashboard, media app, learning app,

- What does the user/player repeatedly do?
- What makes the app interesting right now?
- What is missing from the experience?

- Genre: arcade, platformer, RPG, shooter, puzzle, survival, strategy, idle,

- Player fantasy: what should the player feel like they are doing?
- Current win/loss conditions.
- Current challenge model.
- Current reward model.
- Current progression model.
- Current content model: levels, maps, enemies, items, obstacles, goals.

## Step 3 — App/Game Systems Renovation Pattern Library

Apply every pattern that fits the target app.

================================================================================
A. Architecture and Data Patterns
================================================================================

[ ] A1. Convert scattered hardcoded values into centralized data registries for
    items, enemies, weapons, armor, quests, maps, levels, abilities, recipes, or
    app-domain entities.

[ ] A2. Introduce a central state object that represents the real app/game state:
    player, world, inventory, progress, active enemies, quests, timers, settings,
    save data, and runtime flags.

[ ] A3. Separate update logic from render logic where applicable. The app should
    be able to update state, then render that state consistently.

[ ] A4. Add stable factories/helpers for creating repeated entities such as
    enemies, projectiles, items, cards, documents, tasks, nodes, resources, or
    other domain objects.

[ ] A5. Add versioned save data and migration hooks when persistent progress is
    introduced.

[ ] A6. Define constants for tuning: damage, speed, spawn rates, XP curves,
    prices, cooldowns, rarity weights, level thresholds, and difficulty scaling.

================================================================================
B. Core Gameplay / Core Workflow Patterns
================================================================================

[ ] B1. Strengthen the core loop so the user/player has a clear cycle of action,

[ ] B2. Add meaningful choices. Different tools, routes, upgrades, items,

[ ] B3. Add short-term, medium-term, and long-term goals.

[ ] B4. Add rewards that change what the user/player can do.

## Step 4 — Generate an Implementation Plan

Based on the assessment and leap analysis, produce an ordered implementation
plan before coding.

The plan must include:

4.1 Every major system, mechanic, content set, or workflow to be added.

4.2 Work grouped into coherent stages, such as:

- Stage 1: Architecture and state model
- Stage 2: Core loop improvements
- Stage 3: Content registries
- Stage 4: Progression systems
- Stage 5: Enemies and AI
- Stage 6: Items, equipment, inventory, and economy
- Stage 7: World, maps, levels, and biomes
- Stage 8: Quests, objectives, and rewards
- Stage 9: Bosses and advanced encounters
- Stage 10: Persistence, replayability, and saves
- Stage 11: Feedback, effects, audio, and game feel
- Stage 12: Verification and documentation

4.3 For each item, note:

- Files touched.
- Systems affected.
- Data structures introduced.
- Patterns implemented.
- Dependencies on earlier work.
- Relative effort: low / medium / high.
- Risk level: low / medium / high.

4.4 Explicitly call out foundational refactors required before adding features.

4.5 Prioritize working features over decorative placeholders.

## Final Reminder

Do not merely make the app prettier.

A successful renovation makes the app itself:

MORE SUBSTANTIAL   — it has more real systems, content, and capability.
MORE PLAYABLE      — if it is a game, it is more fun moment to moment.
MORE REPLAYABLE    — there are reasons to come back.
MORE PROGRESSIVE   — users/players grow, unlock, discover, or improve.
MORE CONNECTED     — systems interact instead of sitting isolated.
MORE RELIABLE      — repeated use, saving, resetting, and edge cases work.
MORE COMPLETE      — features are functional, not decorative.
MORE MEMORABLE     — the app has moments, surprises, rewards, and identity.

If the original renovation prompt helped improve the GUI, this prompt should
help improve the actual experience, mechanics, systems, and content.
